🗃️ Persistencia, Aislamiento y Principio de Mínimo Privilegio

Los contenedores son efímeros por diseño. Para datos críticos, aislamiento real y endurecimiento, Compose ofrece tres bloques fundamentales:

  • Volúmenes: Nombrados (gestión Docker), Bind Mounts (ruta host) y tmpfs (RAM). Solo los nombrados sobreviven a docker compose down.
  • Redes: Por defecto todos comparten _default. Definir redes propias permite aislar servicios. internal: true corta salida a Internet.
  • Seguridad: read_only: true bloquea escritura en raíz. cap_drop: ["ALL"] quita privilegios Linux. mem_limit y cpus evitan fugas de recursos.

💡 Clave del PDF: Combina read_only con tmpfs en /tmp o /var/run para que la app funcione sin romper la política de solo lectura.

🎯 Reto: Endurece un servicio web

Modifica el bloque para cumplir:

  1. Sistema de ficheros raíz solo lectura (read_only)
  2. Elimina todas las capacidades Linux (cap_drop)
  3. Añade un tmpfs para /tmp
  4. Define una red interna con internal: true
  5. Límites: mem_limit: 256m y cpus: "0.5"

🔍 Checklist de Despliegue Seguro

  • ✅ ¿Usa read_only: true? Si falla, añade tmpfs en /tmp o /var/run
  • ✅ ¿Incluye cap_drop: ["ALL"]? Solo añade cap_add si es estrictamente necesario
  • ✅ ¿Las BBDD o workers están en redes internal: true?
  • ✅ ¿Tiene mem_limit y cpus definidos?
  • ✅ ¿Evita privileged: true? (Casi nunca es necesario)

💡 Valida: docker compose config --quiet y revisa logs con docker logs para detectar permisos denegados por read_only.
