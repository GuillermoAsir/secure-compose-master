🌱 De un contenedor simple a una aplicación real

Un docker-compose.yml define servicios bajo la clave services:. Cada servicio es un contenedor independiente con su propia imagen, red y almacenamiento.

  • image: Imagen pública o privada. Nunca uses :latest en producción.
  • ports: Expone al host (HOST:CONTAINER). expose solo comunica entre servicios de la misma red.
  • environment: Variables de entorno. Acepta lista (- KEY=v) o mapa (KEY: v).
  • env_file: Carga variables desde .env. Ideal para credenciales.
  • volumes: Persistencia. Nombrados (gestión Docker), bind mounts (ruta host) o tmpfs (RAM).

💡 Del PDF: Las dos sintaxis de environment son válidas, pero la de mapa permite mejor resaltado. Ojo: en mapa, DEBUG: false se parsea como booleano. Usa DEBUG: "false".

environment: Lista vs Mapa

# Lista (KEY=valor)
environment:
  - DEBUG=false
  - DB_PORT=${DB_PORT:-5432}

# Mapa (clave: valor)
environment:
  DEBUG: "false"          # Comillas obligatorias
  DB_PORT: ${DB_PORT:-5432}

🔍 Checklist de Buenas Prácticas

  • ✅ Fija versiones de imágenes (nginx:1.27-alpine, no :latest)
  • ✅ Usa env_file o interpolación ${VAR} para secretos
  • ✅ Solo expone con ports: lo que realmente necesita el host
  • ✅ En environment tipo mapa, envuelve booleanos/strings reservados en comillas
  • ✅ Declara volumes: a nivel raíz para volúmenes nombrados

💡 Valida siempre: docker compose config --quiet antes de up.
