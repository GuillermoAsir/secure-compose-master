⚙️ Controlando el arranque y la resiliencia

Un contenedor no solo "se ejecuta". Necesita órdenes claras sobre cómo iniciar, qué hacer si falla y cuándo está realmente listo.

  • command / entrypoint: Sobrescriben el CMD y ENTRYPOINT del Dockerfile.
  • restart: Define la política de resurrección automática.
  • healthcheck: Comando que Docker ejecuta periódicamente para verificar que el servicio responde.
  • depends_on: Garantiza orden de arranque. Combinado con condition: service_healthy, evita errores de conexión temprana.
  • init: Si es true, inyecta un proceso tini para reapear zombies y propagar señales correctamente.

💡 Clave del PDF: depends_on corto solo garantiza que el contenedor arranca antes. Para esperar a que esté listo, usa healthcheck + condition: service_healthy.

Políticas de restart

ValorComportamientoUso recomendado
noNunca reiniciaDesarrollo, jobs puntuales
alwaysReinicia siempre (incluso si lo paras tú)⚠️ Peligroso en producción
on-failure[:N]Solo si sale con código != 0Workers, tareas reintentables
unless-stoppedReinicia siempre, respeta tu stop manual✅ Producción (recomendado)

Parámetros de healthcheck

ClaveFunción

timeoutTiempo máximo antes de marcar fallo (ej: 5s)
